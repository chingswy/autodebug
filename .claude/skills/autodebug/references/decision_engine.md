# 决策引擎：ReAct (推理-行动)

替代旧版"按优先级选第一个未尝试方向"的被动逻辑。

---

## ReAct 循环流程

```
                    ┌──────────────────┐
                    │  加载最新状态    │
                    │  history + insights │
                    └────────┬─────────┘
                             ▼
                 ┌───────────────────────┐
                 │  2.1 观察综合         │
                 │  读中间输出 + 对比    │
                 │  已有结论 → 识别新异常│
                 │  + 检查基础设施状态   │
                 └────────┬──────────────┘
                          ▼
                 ┌───────────────────────┐
                 │  2.2 假设更新         │
                 │  对异常生成因果假设   │
                 │  交叉检查失败/成功    │
                 │  → 排序后活跃假设列表 │
                 └────────┬──────────────┘
                          ▼
                 ┌───────────────────────┐
                 │  2.3 行动选择         │
                 │  按优先级梯度选择     │
                 │  P1 > P2 > P3 > P4   │
                 └────────┬──────────────┘
                          ▼
                 ┌───────────────────────┐
                 │  2.4 预注册           │
                 │  写入假设+预期+检验   │
                 │  指标 (写入 history)  │
                 └───────────────────────┘
```

## 2.1 观察综合

1. 读最新 history 条目的"中间过程观察"和"执行结果"
2. 对比 insights.md "过程观察结论"中的已有认知
3. 识别**新异常**：
   - 之前没观察到的新模式
   - 与预期不符的指标变化
   - 成功/失败修改之间的差异模式
4. 审视已有结论：哪些是经过实验确认的事实，哪些只是"觉得应该是这样"？后者不能作为后续推理的前提
5. 关联多次迭代记录，寻找跨迭代的规律——单次看不出的模式，多次对比往往能发现
6. **检查基础设施状态**（读 insights.md "基础设施状态" 节）：
   - 快速验证脚本是否已创建？
   - 数据是否已分析分类？（若问题涉及多组数据/case）
   - baseline 是否已建立？
   - 基础设施缺口影响 P2 优先级判断

## 2.2 假设更新

对每个新异常和既有未解问题，生成或更新因果假设。

**关键心态**：永远不要对自己的假设深信不疑。调了很久还没找到原因时，要反过来质疑最基础的前提——测试数据对不对？观测工具本身有没有 bug？

### 假设生成的 5 种来源

**1. 矛盾分析** — 预测 vs 实际不符
> "我预期 X 会改善指标 A，但实际恶化了。可能原因：..."
> 产生假设：H_n: X 的效果被因素 Y 抵消

**2. 根因追溯** — 指标变化 → 沿代码路径反向追踪
> "指标 A 恶化 → A 由函数 f() 计算 → f() 的输入来自 g() → g() 在迭代 M 被修改"
> 产生假设：H_n: g() 的修改产生了副作用

**3. 失败分解** — "想法错了还是实现错了？"
> 方向 D 尝试了 3 次都失败 → 分解：
> - H_a: 方向 D 本身就不对（底层假设有误）
> - H_b: 方向 D 正确但实现细节有误（参数/位置/条件）
> 需要不同的验证手段来区分 H_a vs H_b

**4. 类比推理** — 成功经验的迁移
> "方向 A 通过机制 M 改善了指标 X。问题 P 也涉及类似的机制 M'。"
> 产生假设：H_n: 将 M 的思路应用于 P 可能有效

**5. 逆向推理** — 排除法定位
> "对组件 C 的所有修改都失败 → 问题可能不在 C，而在流入 C 的数据"
> 产生假设：H_n: 问题在 C 的上游组件

### 假设排序

按置信度排序，置信度由以下因素决定：
- 支持证据数量和强度
- 反面证据
- 与已证实假设的一致性
- 验证成本（低成本假设优先，因为信息价值/成本比高）

## 2.3 行动选择（优先级梯度）

每次迭代只做一个最小修改，确认其单独影响后再进入下一步。改多了分不清是哪个改动的锅。

```
P1: CRASH 修复
    触发: exit code != 0, traceback, SyntaxError, ImportError, 编译错误
    行动: 原地修复语法/导入/编译错误
    计数: 不计为失败方向，history 标记为 "CRASH FIX"
    上限: 最多连续重试 2 次，超过则回退代码

P2: 基础设施 + 观测缺口
    触发条件（任一满足即触发）:
    ├─ 无快速验证脚本 → 创建精简验证脚本（语言和形式由项目决定）
    ├─ 未分析数据 → 逐条运行并分类（🔴🟡🟢）（若问题涉及多组数据/case）
    ├─ 无 baseline → 对重点 case 建立 baseline 指标
    ├─ 需要新观测指标 → 实现并添加到测试脚本
    └─ 最高置信假设无法用当前 instrumentation 验证 → 添加针对性日志/观测
    目标: 确保有足够的基础设施和观测能力来验证假设
    优先级内部排序: 验证脚本 > 数据分析 > baseline > 观测缺口
    history 标记: "INFRA" (基础设施) 或正常迭代

P3: 假设驱动修改
    触发: 有充分观测数据的最高置信假设
    行动: 设计最小修改来验证假设
    要求: 必须预注册预期结果

P4: 探索性
    触发: 无强假设，所有假设置信度都低
    行动: 从以下来源生成新方向
      - brief.md 方向列表中未尝试的
      - 代码分析发现的新可能
      - 参考文献知识中的方法
    要求: 即使是探索性也要明确写出假设
```

## 首次迭代引导

Startup 完成后，第一次进入 ReAct 循环时的自然行为：

1. **2.1 观察综合**：无历史迭代可分析，但检查基础设施状态 → 发现全部为 ❌
2. **2.2 假设更新**：可能从代码知识库和用户描述生成初步假设，但无观测数据支撑
3. **2.3 行动选择**：P2 触发 — 无快速验证脚本，优先创建
4. **执行**：创建验证脚本，对测试数据逐条运行，收集指标
5. **记录**：作为 Iteration 1 (INFRA) 记录到 history.md

后续迭代自然继续：
- 第 2 次迭代：P2 触发 — 有验证脚本但无数据分类 → 分析数据并分类（若适用）
- 第 3 次迭代：P2 触发 — 有分类但无 baseline → 建立 baseline（若适用）
- 第 4+ 次迭代：基础设施就绪，P2 只在需要新指标时触发，否则进入 P3/P4

这个过程不是固定的——如果在创建测试脚本时就发现了明显的 crash，P1 会立刻介入；
如果用户描述的问题直接指向已知根因，可能跳过部分基础设施直接进入 P3。
ReAct 的优先级机制保证了正确的事情在正确的时机发生。

## 2.4 预注册

在修改代码前，写入 history 当前迭代的"假设与预注册"部分。不允许跳过验证直接改——猜测只用于缩小范围，确认根因必须靠实验。

```markdown
### 假设与预注册
- 假设: <本次验证的因果假设 ID 和描述>
- 预期结果: <如果假设成立，哪个指标应该如何变化>
- 检验指标: <具体看哪个数值，阈值是多少>
- 如果与预期不符: <说明什么？排除了什么可能？>
```

迭代完成后，对比预测 vs 实际，写入"预测 vs 实际"部分。这个反馈循环驱动假设置信度的更新。

> **注意**：对于 INFRA 类型迭代，预注册可简化为"目标 + 完成标准"，无需假设验证格式。

## 自动 Pivot 规则

| 触发条件 | 行动 |
|---------|------|
| 同组件连续 3 次失败 | 标记为"已充分探索"，强制转向其他组件 |
| instrumentation 揭示根因在其他组件 | 立即重定向到新组件 |
| 成功修改暴露新瓶颈 | 记录转折点，追踪新瓶颈 |
| 连续 3 次无改善且无新信息 | 降低当前方向簇优先级，转向 |

执行 pivot 时，在 insights.md "策略转折点" 章节记录：
```markdown
- Iter N: 从 <A> 转向 <B>，原因：<观察>
```

## 增强版反循环机制

### 基础检查（同旧版）
- insights.md "失败尝试" 记录**具体做法**（含参数值、修改位置）
- 选方向前对比失败记录 + 近 5 条 history，实质相同的尝试跳过

### 增强检查（新增）
- **语义去重**：不只比较方向名，还比较修改的代码位置和逻辑。"调整 A 参数为 0.1" 和 "调整 A 参数为 0.2" 是同一方向不同参数，连续 3 次参数调整无效则标记整个方向
- **模式检测**：如果最近 3 次迭代的假设→结果模式相同（如"修改 X → 指标无变化"），识别为循环
- **自动升级**：检测到循环 → 强制执行探索性行动或 pivot

### 退出条件
- 所有方向被排除且无法生成新方向 → `status: done`
- 目标指标已达成 → `status: done`
- 连续 5 次迭代无任何改善或新信息 → `status: done`
