# Startup 协议（轻量启动）

首次运行时 SKILL.md 指向本文件。仅执行用户交互和代码理解，然后直接进入 `status: running`。
基础设施搭建（数据分析、测试脚本、baseline）由后续 ReAct 迭代自然完成。

---

## Startup — 用户交互 + 知识库构建

**目标**：获取 Agent 无法自己获取的领域知识，理解代码，然后立即进入迭代。

### 步骤

1. **快速询问用户**（使用 AskUserQuestion 工具，一次最多 4 个问题）：
   - "你观察到的具体现象是什么？"
   - "是否有参考文献/论文/代码可以参考？（URL 或本地路径）"
   - "测试数据中哪些 case 是最需要关注的？哪些有共性问题可以暂时忽略？"
   - "你期望的目标指标是什么？有没有量化标准？"

2. **参考文献处理**（如果用户提供了）：
   - 启动 Agent 子代理阅读参考文献
   - 提炼与调试相关的方法论知识
   - 写入 brief.md "参考文献知识" 章节

3. **构建代码知识库**（写入 brief.md "代码知识库" 章节）：
   - 阅读项目文档体系：从项目根目录的文档（如 README、CLAUDE.md、docs/ 等）开始，沿模块索引读取相关子模块文档
   - 阅读代码入口文件：读取 brief.md "代码入口" 列出的文件
   - 追踪引用链：从入口文件出发读取被 import 的关键模块（只追踪与问题相关的）
   - 提炼并写入：
     - `### 项目架构摘要`
     - `### 关键文件索引`
     - `### 核心逻辑摘要`
     - `### 领域知识备忘`

4. **分析问题的可观测性**：用户描述的问题可能无法直接通过代码输出观测（如视觉问题、性能问题、非确定性bug），需要思考如何通过可量化的指标间接反映。将观测方案写入 brief.md "Observability Mapping" 和 insights.md "当前观测手段"

5. 创建 `$WORKSPACE/history.md`（参考 `references/memory_formats.md` 头部格式，重点数据标注"首次迭代后填充"）

6. 创建 `$WORKSPACE/insights.md`，直接设 `status: running`

7. 输出 Startup 摘要并退出

---

## Startup 完成后的自然过渡

Startup 完成后，insights.md 已是 `status: running`，下次调用直接进入 ReAct 循环。

第一次进入 ReAct 时，decision_engine.md 的 P2 优先级会自然检测到基础设施缺口：
- 无快速验证脚本 → 创建精简验证脚本
- 未分析数据 → 逐条运行并分类（若适用）
- 无 baseline → 建立 baseline 指标（若适用）

这些都是有效迭代（类型 E: 基础设施搭建），正常记录到 history.md。
不需要特殊的初始化阶段，ReAct 的优先级机制自然驱动这些工作发生。

---

## 设计理由

- **只做一件事**：Startup 只获取 Agent 无法自主获取的信息（用户知识 + 代码理解）
- **基础设施融入迭代**：数据分析、测试脚本、baseline 都是可以自主完成的工作，放在 ReAct 迭代中更自然
- **消除人为边界**：不再有"初始化完成才能开始调试"的割裂感，Startup 后立即可以迭代
- **动态灵活**：后续迭代中随时可以添加新指标、分析新数据，不受"初始化已完成"的限制
