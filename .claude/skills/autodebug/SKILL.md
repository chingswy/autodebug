# Autodebug Skill

假设驱动的迭代式自动调试。每次调用完成一个 ReAct (推理-行动) 迭代。
核心设计：轻量 Startup（仅用户交互+代码理解）后立即进入 ReAct 迭代循环，
基础设施搭建（测试脚本、数据分析、baseline、观测指标）作为自然迭代由 P2 优先级驱动。

## 核心原则

- **添加有效观测指标 = 有效迭代**：创建验证脚本、分析数据、建立 baseline、添加新指标都是有效的迭代工作，不是"准备工作"
- **动态调整思考维度**：迭代过程中随时可以发现需要新指标就加、发现代码可疑就加日志，不受阶段限制
- **基础设施融入迭代**：没有"初始化完成"的概念，一切都是迭代

## 使用方式

- `/autodebug /path/to/workspace` — 在指定 workspace 执行一次迭代
- 外部循环：`python .claude/skills/autodebug/run_loop.py --workspace /path/to/workspace`

## Workspace 文件

| 文件 | 用途 | 模式 |
|------|------|------|
| `brief.md` | 问题背景+方向+命令+数据分析+代码知识库 | 首次自动填充，人类可编辑 |
| `history.md` | 每次迭代的完整记录 | 追加 |
| `insights.md` | 知识+假设+方向+退出信号+基础设施状态 | 每次安全重写 |
| `test_quick.*` | 精简快速验证脚本（首次迭代中创建，语言和形式由项目决定） | skill 创建和维护 |
| `backups/iter_N/` | 每次迭代的文件备份 | 自动创建 |

模板：`references/brief_template.md` | 记忆格式：`references/memory_formats.md`

## 调试思想

以下不是检查清单，而是内化到每个阶段的行为准则：

- **不理解就不动手。** 改代码之前，必须能说清目标模块的输入、输出和数据流。读不懂的代码恰恰是 bug 最可能藏身的地方。宁可多花时间读代码，也不要在不理解的情况下瞎改。
- **用事实说话，不靠猜。** 猜测只用于缩小搜索范围；定位根因必须靠实验观察。没有亲眼看到底层细节如何引发 bug，就不要修。盲目修复常常制造新问题。
- **每次只改一个东西。** 控制变量才能确认因果。一次改多处，出了问题分不清是哪处的锅。
- **不断缩小范围。** 用二分法逼近 bug：排除正常的一半，聚焦异常的一半。用 toy data 和最小配置隔离问题，减少多个 bug 同时出现的干扰。
- **对自己的假设保持怀疑。** 尤其是调了很久还没找到原因的时候，要反过来质疑最基础的前提——测试数据对不对？观测工具本身有没有 bug？
- **记录驱动发现。** 每次迭代的做法、结果、中间观察都要记录。跨迭代关联这些记录，往往能发现单次看不出的规律。

## 执行流程

`$ARGUMENTS` 是 workspace 目录路径。如果为空，向用户询问。

### Phase 1: 加载状态

1. 设置 `WORKSPACE=$ARGUMENTS`
2. **检查残留文件**：如果存在 `insights.md.new` 或 `insights.md.bak`，执行恢复：
   - `.new` 存在但正式文件损坏 → rename `.new` 为正式
   - `.bak` 存在 → 删除 `.bak`
3. 读取 `$WORKSPACE/brief.md` — 如果不存在 → 报错退出，提示用户参照模板创建
4. 读取 `$WORKSPACE/insights.md`（全量）
   - 检查 `status:` 字段：`done` → 输出"调试已完成"并退出
   - **兼容性**：任何非 `done` 的 status（包括旧版 `initializing`）都按 `running` 处理
5. 读取 `$WORKSPACE/history.md`（仅最后 5 条 `## Iteration` 块）
6. 判断路由：
   - `insights.md` 不存在 → **Startup Phase**
   - `insights.md` 存在且 status 非 `done` → **Phase 2 ReAct**

### Startup Phase

详细步骤见 `references/startup_protocol.md`。

仅执行用户交互和代码理解：
- 用 AskUserQuestion 询问：问题现象、参考文献、重点数据、目标指标
- 如有参考文献 → 启动 Agent 子代理阅读并提炼
- 构建代码知识库（写入 brief.md）——这一步的目的是真正理解代码，后续才有能力判断什么是正常什么是异常
- 分析问题可观测性：难以直接观测的问题 → 可量化指标映射
- 创建 history.md + insights.md (`status: running`)
- 输出 Startup 摘要并退出

### Phase 2: ReAct 决策循环

详细流程见 `references/decision_engine.md`。

**2.1 观察综合：**
- 读最新 history 的中间观察，对比 insights 已有结论
- 识别**新异常**：之前没观察到的模式、与预期不符的变化
- 检查：当前结论是否每条都有事实支撑？是否有"觉得应该是这样"但未经验证的认知混入？
- **检查基础设施状态**：快速验证脚本 / 数据分类 / baseline 是否就绪？缺口影响 P2

**2.2 假设更新：**
- 对异常生成因果假设（5 种来源：矛盾分析、根因追溯、失败分解、类比推理、逆向推理）
- 交叉检查失败/成功记录
- 输出排序后的**活跃假设列表**
- 审视假设列表：有没有哪个假设已经隐含地被当成了事实？如果有，标记为"需验证"

**2.3 行动选择（优先级梯度）：**
```
P1: CRASH 修复 — 语法/导入/编译错误 → 原地修复，不计为失败方向（最多重试 2 次）
P2: 基础设施 + 观测缺口 — 无快速验证脚本/未分析数据/无baseline/需要新指标/假设无法验证 → 搭建或补充
P3: 假设驱动修改 — 有充分观测的最高置信假设 → 最小修改验证
P4: 探索性 — 无强假设 → 从 brief.md/代码分析/文献生成新方向
```

**2.4 预注册：**
- 修改前写入：假设、预期结果、检验指标
- 必须回答"如果实验结果和预期不符，说明什么？"——这迫使你在动手前想清楚
- **INFRA 迭代**：预注册可简化为"目标 + 完成标准"

### Phase 3: 修改代码

1. 从 brief.md "代码知识库" 获取上下文（不足时才读源文件，读后补充回 brief.md）
2. 确认自己理解目标模块：能回答"这段代码正确执行时，输出应该是什么"。如果答不上来，先读代码直到能回答
3. **备份**：`$WORKSPACE/backups/iter_N/` — `mkdir -p` + `cp` 保留相对路径
4. **最小修改**：instrumentation → 添加 print；逻辑修改 → 精确修改一处目标代码，不夹带任何额外改动
5. 用 Edit 工具执行修改，记录改动

### Phase 4: 执行 + 分类

**精简测试策略：**
- 迭代中 → 先用快速验证脚本对最具代表性的 case 验证
- 单条通过 → 再跑完整重点 case 集合
- 即将 `status: done` 前 → 跑全量验证

**执行规则：**
- Bash timeout：默认 300000ms；长时间任务可酌情延长至 600000ms
- 先检查 exit code + traceback：
  - CRASH 检测 → 快速修复协议（最多重试 2 次）
  - 正常 → 按"结果解析"提取指标

**结果分类：**

| 情况 | 处理 |
|------|------|
| A: 改善 | 保留修改，标记有效，更新 best_result |
| B: 恶化/报错 | 从 backup 恢复，标记失败+原因 |
| C: 无变化但有新信息 | 保留修改，记录观察 |
| D: 纯 instrumentation | 保留修改，不评判指标 |
| E: 基础设施搭建 | 保留产出，记为有效迭代，不评判指标改善 |

### Phase 5: 记录

**5a. 追加 history.md** — 格式见 `references/memory_formats.md`
- 含：假设与预注册、方向、修改、执行结果、预测vs实际、结论、回退
- **INFRA 迭代**使用基础设施迭代条目模板

**5b. 安全重写 insights.md：**
```
写 insights.md.new → rename 旧→.bak → rename .new→正式 → 删 .bak
```
- 更新：基础设施状态、活跃假设列表、观测手段、过程结论、有效/失败方向、转折点、数据验证状态、待探索方向
- 格式见 `references/memory_formats.md`

**5c. 输出迭代摘要**

## 主动性规则

**每轮迭代后必须尝试生成新假设：**
1. 矛盾分析 — 预测 vs 实际不符，为什么？
2. 根因追溯 — 指标变化 → 沿代码路径反向定位
3. 失败分解 — "想法错了还是实现错了？" → 拆出子假设
4. 类比推理 — "方向 A 的机制 M 能否用在问题 P 上？"
5. 逆向推理 — "组件 C 全失败 → 问题可能在 C 的上游"

**自动 Pivot 规则：**
- 同组件连续 3 次失败 → 标记"已充分探索"，强制转向
- instrumentation 揭示根因在其他组件 → 立即重定向
- 成功修改暴露新瓶颈 → 记录转折点，追踪新瓶颈
- 连续 3 次无改善且无新信息 → 降低当前方向簇优先级

## 鲁棒性规则

**insights.md 安全写入：** 写 `.new` → rename 旧→`.bak` → rename `.new`→正式 → 删 `.bak`。Phase 1 检查残留并恢复。

**快速失败协议：** 运行后先检查 exit code + traceback/SyntaxError/ImportError/编译错误 → CRASH 原地修复（最多 2 次）→ 不算失败方向 → history 标记为 "CRASH FIX"。

**Bash timeout：** 默认 300000ms (5min)，长时间任务 600000ms (10min)。超时 → 记为 TIMEOUT，考虑更轻量验证。

---

## 反循环机制

- 失败尝试记录**具体做法**（参数值、修改位置），不只方向名
- 选方向前对比失败记录 + 近 5 条 history，实质相同跳过
- **语义去重**：连续 3 次参数调整无效 → 标记整个方向
- **模式检测**：最近 3 次假设→结果模式相同 → 强制 pivot
- 所有方向排除且无法生成新方向 → 自动退出

## 退出条件

以下任一满足，写 `status: done` 到 insights.md：
1. 目标指标已达成
2. 待探索方向为空且无法生成新方向
3. 连续 5 次迭代无任何改善或新信息

## 重要约束

- **知识库优先**：后续迭代优先从 brief.md "代码知识库" 获取上下文，避免反复读源文件
- **遵守目录安全规则**：不对数据目录递归搜索
- **最小修改原则**：每次只改必要的代码，不顺手重构
- **不自动 commit**：代码修改不自动提交 git
- **禁止 git checkout 回退**：回退必须使用备份文件恢复
- **修改前备份**：备份到 `$WORKSPACE/backups/iter_N/`
- **不使用 try/except 掩盖错误**：错误要暴露出来，便于定位问题
- **中文输出**：迭代摘要、记忆文件使用中文
